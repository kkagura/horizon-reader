# Horizon Reader 技术方案

## 1. 当前项目定位

本项目用于构建一个基于 Horizon 的 AI 新闻日报阅读系统。

当前仓库已经包含 Horizon 的服务器部署资料，但还没有实现 Nuxt Markdown 阅读服务。

整体边界如下：

```text
Horizon
  负责采集新闻、AI 筛选、总结、生成 Markdown

Horizon Reader
  负责读取 Markdown、渲染页面、提供归档和检索
```

关键原则：

- Horizon 不嵌入 Web 请求链路。
- Web 阅读器不负责抓取新闻，不直接调用 AI 模型。
- Horizon 通过 Docker 镜像在服务器上按需运行。
- Horizon 生成的 Markdown 写入 `reports/`，阅读器运行时读取该目录。
- Markdown 新增、修改、删除后，阅读器不需要重新构建。

## 2. 当前目录结构

当前实际目录结构：

```text
horizon-reader/
  技术方案.md

  horizon-deploy/
    build-image.ps1
    server-run.sh
    env.example
    config.example.json
    README.md
    .gitignore

    .env                  # 生产环境密钥文件，已忽略
    data/
      config.json         # 生产环境 Horizon 配置，已忽略
```

说明：

- `horizon-deploy/build-image.ps1`：本地构建 Horizon Docker 镜像并导出 `horizon-image.tar`。
- `horizon-deploy/server-run.sh`：服务器上导入镜像、运行 Horizon、复制 Markdown 到 `reports/`。
- `horizon-deploy/env.example`：环境变量模板，不包含真实密钥。
- `horizon-deploy/config.example.json`：Horizon 配置模板。
- `horizon-deploy/.env`：生产环境环境变量，包含真实 API Key，不应提交仓库。
- `horizon-deploy/data/config.json`：生产环境 Horizon 配置，不应提交仓库。

当前不采用 pnpm monorepo，也不把 Horizon 源码纳入本仓库。Horizon 源码只在本地构建镜像时临时拉取到 `.horizon-build/`，该目录已忽略。

## 3. Horizon 部署方案

采用“本地构建镜像，服务器单脚本运行”的方式。

本地构建：

```powershell
cd E:\wang\horizon-reader\horizon-deploy
.\build-image.ps1
```

构建脚本会：

1. 克隆或更新 Horizon 官方源码到 `.horizon-build/Horizon`。
2. 使用 Horizon 官方 Dockerfile 构建镜像。
3. 导出镜像为 `horizon-image.tar`。

上传到服务器的文件：

```text
horizon-image.tar
server-run.sh
.env
data/config.json
```

服务器运行：

```bash
cd /opt/horizon-deploy
chmod +x server-run.sh
./server-run.sh
```

服务器目录建议：

```text
/opt/horizon-deploy/
  horizon-image.tar
  server-run.sh
  .env
  data/
    config.json
    summaries/
  reports/
```

运行结果：

- Horizon 原始输出位于 `/opt/horizon-deploy/data/summaries/`。
- 给阅读器读取的 Markdown 位于 `/opt/horizon-deploy/reports/`。

如果要每天自动运行，可手动添加 cron：

```cron
0 8 * * * cd /opt/horizon-deploy && ./server-run.sh >> horizon.log 2>&1
```

- 不要把 `data/config.json` 放到 Web 静态目录。

## 5. 后续阅读器推荐技术栈

阅读器建议使用：

```text
Nuxt 3 + Vue 3 + TypeScript + pnpm
```

推荐依赖：

```bash
pnpm add markdown-it gray-matter fast-glob
pnpm add -D @types/markdown-it
```

可选依赖：

```bash
pnpm add minisearch
pnpm add sanitize-html
```

用途：

- `gray-matter`：解析 Markdown frontmatter。
- `markdown-it`：Markdown 转 HTML。
- `fast-glob`：扫描 Markdown 文件。
- `minisearch`：后续实现本地全文搜索。
- `sanitize-html`：公开访问场景下清洗 HTML。

## 6. 阅读器运行模式

必须采用 Nuxt SSR / Node server 模式。

开发：

```bash
pnpm dev
```

生产：

```bash
pnpm build
node .output/server/index.mjs
```

不要把纯静态生成作为默认部署方式：

```bash
nuxi generate
```

原因：

- 静态生成会在构建时固化 Markdown 内容。
- Horizon 后续生成新 Markdown 后，静态站点需要重新生成才能看到新内容。
- SSR / Node server 模式可以在请求时读取最新 Markdown 文件。

SSR 模式下：

- 新增 Markdown：不需要重新构建。
- 修改 Markdown：不需要重新构建。
- 删除 Markdown：不需要重新构建。
- 修改 Nuxt 页面、组件、服务端代码：需要重新构建。

## 8. Markdown 文件规范

推荐文件命名：

```text
YYYY-MM-DD.zh.md
YYYY-MM-DD.en.md
```

示例：

```text
2026-05-25.zh.md
2026-05-25.en.md
```

slug 规则：

- `2026-05-25.zh.md` 的 slug 为 `2026-05-25.zh`。
- URL 为 `/reports/2026-05-25.zh`。

推荐 frontmatter：

```markdown
---
title: "AI 新闻日报"
date: "2026-05-25"
lang: "zh"
source: "horizon"
---

# AI 新闻日报
```

如果 Horizon 输出没有 frontmatter，服务端从文件名推导：

- `date` 从文件名前 10 位读取。
- `lang` 从 `.zh` 或 `.en` 读取。
- `title` 默认使用 `YYYY-MM-DD 中文日报` 或 `YYYY-MM-DD English Daily`。

## 9. 服务端 API 设计

### 9.1 获取日报列表

```http
GET /api/reports
```

响应：

```ts
interface ReportListItem {
  slug: string
  title: string
  date: string
  lang: 'zh' | 'en' | string
  excerpt?: string
  updatedAt: string
}
```

要求：

- 扫描 `REPORTS_DIR` 指向的目录。
- 只返回元数据，不返回完整正文。
- 忽略非 `.md` 文件。
- 默认按日期倒序。

### 9.2 获取日报详情

```http
GET /api/reports/:slug
```

响应：

```ts
interface ReportDetail {
  slug: string
  title: string
  date: string
  lang: string
  markdown: string
  html: string
  updatedAt: string
}
```

要求：

- 根据 slug 映射到 `REPORTS_DIR/{slug}.md`。
- 必须防止路径穿越。
- 文件不存在返回 404。
- Markdown 在服务端渲染。

## 10. 服务端实现约束

`server/utils/reports.ts` 负责扫描和读取 Markdown 文件。

建议接口：

```ts
export interface ReportMeta {
  slug: string
  title: string
  date: string
  lang: string
  excerpt?: string
  updatedAt: string
}

export interface ReportDetail extends ReportMeta {
  markdown: string
  html: string
}

export async function listReports(): Promise<ReportMeta[]>
export async function getReportBySlug(slug: string): Promise<ReportDetail | null>
```

`server/utils/markdown.ts` 负责 Markdown 解析和渲染。

建议接口：

```ts
export interface ParsedMarkdown {
  data: Record<string, unknown>
  content: string
  html: string
  excerpt?: string
}

export function renderMarkdown(input: string): ParsedMarkdown
```

必须做到：

- 使用 Node `fs/promises` 读取文件。
- 使用 `path.resolve` 后校验目标路径必须位于 `REPORTS_DIR` 内。
- 不把用户输入直接拼接成文件路径。
- 不在 API 中运行 Horizon。
- 不在 Web 请求中执行外部命令。

## 11. 前端页面设计

首页 `/`：

- 展示所有日报。
- 支持按语言筛选。
- 支持按关键词过滤标题和摘要。
- 点击日报进入详情页。

详情页 `/reports/[slug]`：

- 展示日报标题、日期、语言。
- 渲染 Markdown 正文。
- 提供返回列表入口。
- 可选提供原始 Markdown 查看入口。

组件建议：

- `ReportList.vue`：日报列表布局。
- `ReportListItem.vue`：单条日报摘要。
- `ReportToolbar.vue`：搜索和语言筛选。
- `ReportViewer.vue`：正文渲染。
- `EmptyState.vue`：空状态。

## 12. Vue 编码规范

必须使用 Vue 3 Composition API 和 TypeScript。

SFC 结构：

```vue
<script setup lang="ts">
</script>

<template>
</template>

<style scoped>
</style>
```

要求：

- 页面组件只负责数据编排和页面组合。
- 列表、条目、工具栏、正文阅读器拆成独立组件。
- 复杂状态或复用逻辑放入 composable。
- props 和 emits 必须有明确 TypeScript 类型。
- 尽量使用 `computed` 派生状态，避免在模板里写复杂表达式。

避免：

- 把所有功能写进 `pages/index.vue`。
- 在模板里做复杂排序、过滤、日期解析。
- 使用 Options API，除非项目已有明确约定。

## 13. 阅读器配置项

在 `nuxt.config.ts` 中通过 runtime config 配置报告目录。

环境变量：

```text
REPORTS_DIR=/opt/horizon-deploy/reports
```

配置示例：

```ts
export default defineNuxtConfig({
  runtimeConfig: {
    reportsDir: process.env.REPORTS_DIR || './reports',
  },
})
```

服务端读取：

```ts
const config = useRuntimeConfig()
const reportsDir = config.reportsDir
```

如果阅读器也使用 Docker 部署，应挂载 Markdown 目录：

```text
/opt/horizon-deploy/reports:/app/reports
```

并设置：

```text
REPORTS_DIR=/app/reports
```

## 14. 安全要求

必须实现：

- API slug 参数校验，只允许字母、数字、短横线、下划线和点。
- 路径解析后确认文件位于 `REPORTS_DIR` 内。
- 不允许通过 API 读取任意本地文件。
- 不把 Horizon 配置、API Key、日志文件暴露为静态资源。

建议实现：

- 对渲染后的 HTML 做清洗。
- 外链添加 `target="_blank"` 和 `rel="noopener noreferrer"`。
- 对读取失败、解析失败做明确错误响应。

## 15. 后续扩展路线

第一阶段：最小可用版本

- 日报列表。
- 日报详情。

第三阶段：内容能力增强

- 使用 MiniSearch 建立本地搜索索引。
- 支持标签、来源、评分字段。
- 支持 RSS 输出。
- 支持收藏和阅读状态。

第四阶段：数据持久化

- 当需要用户状态、全文索引、访问统计时，再引入 SQLite。
- SQLite 不替代 Markdown 原文，Markdown 仍作为可归档内容源。

# Horizon Markdown Reader 技术方案

## 1. 项目目标

本项目用于展示 Horizon 自动生成的 Markdown 新闻日报。

Horizon 负责新闻采集、AI 筛选、总结和 Markdown 文件生成；本项目只负责读取这些 Markdown 文件，并通过 Web 页面展示、检索和归档。

核心原则：

- Horizon 是离线生成任务，不嵌入 Web 请求链路。
- Web 服务只读取已生成的 Markdown 文件，不负责抓取新闻。
- 新增、修改、删除 Markdown 文件后，Web 服务应能在运行时读取最新内容，不需要重新构建。
- 初始版本不引入数据库，优先使用文件系统作为内容存储。

## 2. 推荐技术栈

使用 Nuxt 3 + Vue 3 + TypeScript + pnpm。

推荐原因：

- Nuxt 同时支持页面和服务端 API，适合构建 Markdown 内容阅读服务。
- Nuxt SSR / Node server 模式可以在运行时读取文件系统。
- Vue 3 Composition API 适合拆分阅读器、列表、筛选和搜索等功能。
- pnpm 依赖管理严格，适合长期维护。

建议依赖：

```bash
pnpm add markdown-it gray-matter fast-glob
pnpm add -D @types/markdown-it
```

可选依赖：

```bash
pnpm add minisearch
pnpm add sanitize-html
```

说明：

- `gray-matter` 用于解析 Markdown frontmatter。
- `markdown-it` 用于 Markdown 转 HTML。
- `fast-glob` 用于扫描日报文件。
- `minisearch` 用于后续本地全文搜索。
- `sanitize-html` 用于公开访问场景下的 HTML 清洗。

## 3. 运行模式

必须采用 Nuxt SSR / Node server 模式。

开发：

```bash
pnpm dev
```

生产构建：

```bash
pnpm build
node .output/server/index.mjs
```

不要使用纯静态生成作为默认部署方式：

```bash
nuxi generate
```

原因是纯静态生成会在构建时固化内容。若 Horizon 后续生成了新的 Markdown 文件，静态站点需要重新生成才能看到新内容。

SSR / Node server 模式下：

- 新增 Markdown 文件：不需要重新构建。
- 修改 Markdown 文件：不需要重新构建。
- 删除 Markdown 文件：不需要重新构建。
- 修改 Nuxt 页面、组件、服务端代码：需要重新构建。

## 4. 总体架构

```text
Horizon 定时任务
  |
  v
reports/*.md
  |
  v
Nuxt Server API
  |
  v
Vue 阅读界面
```

职责划分：

- Horizon：采集新闻、AI 评分、去重、总结、生成 Markdown。
- 定时任务：按计划运行 Horizon，并把输出写入 `reports/`。
- Nuxt Server API：扫描、解析、读取 Markdown 文件。
- Vue 页面：展示日报列表、日报详情、搜索和筛选入口。

## 6. Markdown 文件规范

推荐文件命名：

```text
YYYY-MM-DD.zh.md
YYYY-MM-DD.en.md
```

示例：

```text
2026-05-25.zh.md
2026-05-25.en.md
```

`slug` 规则：

- `2026-05-25.zh.md` 的 slug 为 `2026-05-25.zh`。
- URL 为 `/reports/2026-05-25.zh`。

推荐 Markdown frontmatter：

```markdown
---
title: "AI 新闻日报"
date: "2026-05-25"
lang: "zh"
source: "horizon"
---

# AI 新闻日报
```

如果 Horizon 输出的 Markdown 没有 frontmatter，服务端需要从文件名推导：

- `date` 从文件名前 10 位读取。
- `lang` 从 `.zh` 或 `.en` 读取。
- `title` 默认使用 `YYYY-MM-DD 中文日报` 或 `YYYY-MM-DD English Daily`。

## 7. 服务端 API 设计

### 7.1 获取日报列表

接口：

```http
GET /api/reports
```

响应：

```ts
interface ReportListItem {
  slug: string
  title: string
  date: string
  lang: 'zh' | 'en' | string
  excerpt?: string
  updatedAt: string
}
```

排序规则：

- 默认按 `date` 倒序。
- 同一天多语言版本按 `zh`、`en` 排列。

实现要求：

- 每次请求扫描 `reports/` 或使用短 TTL 缓存。
- 只返回元数据，不返回完整正文。
- 忽略非 `.md` 文件。
- 文件名不符合规范时可以忽略，或返回为 `lang: "unknown"`。

### 7.2 获取日报详情

接口：

```http
GET /api/reports/:slug
```

响应：

```ts
interface ReportDetail {
  slug: string
  title: string
  date: string
  lang: string
  markdown: string
  html: string
  updatedAt: string
}
```

实现要求：

- 根据 slug 映射到 `reports/{slug}.md`。
- 必须防止路径穿越，例如 `../`。
- 如果文件不存在，返回 404。
- Markdown 渲染在服务端完成。
- 如果启用 HTML 清洗，应在返回前处理。

## 8. 服务端实现约束

`server/utils/reports.ts` 负责文件扫描和读取。

建议函数：

```ts
export interface ReportMeta {
  slug: string
  title: string
  date: string
  lang: string
  excerpt?: string
  updatedAt: string
}

export interface ReportDetail extends ReportMeta {
  markdown: string
  html: string
}

export async function listReports(): Promise<ReportMeta[]>
export async function getReportBySlug(slug: string): Promise<ReportDetail | null>
```

`server/utils/markdown.ts` 负责 Markdown 解析。

建议函数：

```ts
export interface ParsedMarkdown {
  data: Record<string, unknown>
  content: string
  html: string
  excerpt?: string
}

export function renderMarkdown(input: string): ParsedMarkdown
```

实现注意事项：

- 使用 Node `fs/promises` 读取文件。
- 使用 `path.resolve` 后校验目标路径必须位于 `reports/` 内。
- 不要把用户输入直接拼接成文件路径。
- 不要在 API 中运行 Horizon。
- 不要在 Web 请求中执行外部命令。

## 9. 前端页面设计

### 9.1 首页 `/`

功能：

- 展示所有日报。
- 支持按语言筛选。
- 支持按关键词过滤标题和摘要。
- 点击日报进入详情页。

组件建议：

- `ReportList.vue`：接收日报数组，负责列表布局。
- `ReportListItem.vue`：展示单个日报的标题、日期、语言、摘要。
- `ReportToolbar.vue`：搜索框和语言筛选。
- `EmptyState.vue`：无内容状态。

数据获取：

```ts
const { data, pending, error, refresh } = await useFetch('/api/reports')
```

### 9.2 详情页 `/reports/[slug]`

功能：

- 展示日报标题、日期、语言。
- 渲染 Markdown 正文。
- 提供返回列表入口。
- 可选：提供原始 Markdown 查看或复制入口。

组件建议：

- `ReportViewer.vue`：接收 HTML 字符串并展示正文。

数据获取：

```ts
const route = useRoute()
const { data, pending, error } = await useFetch(`/api/reports/${route.params.slug}`)
```

注意：

- 如果使用 `v-html` 渲染服务端返回的 HTML，必须确认 HTML 已经由可信 Markdown 渲染器处理。
- 若未来允许外部用户提交 Markdown，必须加 HTML 清洗。

## 10. Vue 编码规范

必须使用 Vue 3 Composition API。

SFC 结构：

```vue
<script setup lang="ts">
</script>

<template>
</template>

<style scoped>
</style>
```

组件原则：

- 页面组件只负责数据编排和页面组合。
- 列表、条目、工具栏、正文阅读器拆成独立组件。
- 复杂状态或可复用逻辑放入 composable。
- props 和 emits 必须有明确 TypeScript 类型。
- 尽量使用 `computed` 派生状态，避免在模板里写复杂表达式。

不建议：

- 把所有功能写在 `pages/index.vue` 一个文件里。
- 在模板里做复杂排序、过滤、日期解析。
- 使用 Options API，除非项目已有明确约定。

## 11. 样式和交互建议

这是一个阅读型工具，界面应偏安静、清晰、适合长期阅读。

建议：

- 首页重点是扫描效率，列表信息密度适中。
- 详情页重点是正文阅读体验，正文最大宽度控制在舒适阅读范围。
- 支持移动端阅读。
- 标题、日期、语言标识要清晰。
- 日报正文样式应覆盖标题、列表、引用、代码块、表格和链接。

避免：

- 做营销式首页。
- 使用大面积装饰性渐变和无关视觉元素。
- 把工具型界面做成卡片堆叠的宣传页。

## 12. Horizon 集成方式

Horizon 应通过定时任务独立运行。

示例流程：

```text
每天 08:00
  运行 Horizon
  输出 Markdown 到 reports/
  Nuxt 服务自动读取新文件
```

Windows PowerShell 示例脚本位置：

```text
scripts/run-horizon.ps1
```

脚本职责：

- 进入 Horizon 项目目录。
- 执行 Horizon 命令。
- 将生成的 Markdown 复制或输出到本项目 `reports/`。
- 写入日志，便于排查失败原因。

重要约束：

- Web 服务不应直接保存 Horizon API Key。
- Horizon 配置文件不应放入 Web 静态目录。
- `reports/` 只存放可公开展示的日报内容。

## 13. 配置项建议

在 `nuxt.config.ts` 中通过 runtime config 配置报告目录。

建议环境变量：

```text
REPORTS_DIR=./reports
```

服务端读取：

```ts
const config = useRuntimeConfig()
const reportsDir = config.reportsDir
```

`nuxt.config.ts` 示例方向：

```ts
export default defineNuxtConfig({
  runtimeConfig: {
    reportsDir: process.env.REPORTS_DIR || './reports',
  },
})
```

如果使用 Docker 部署，应将宿主机目录挂载到容器内：

```text
./reports:/app/reports
```

## 14. 安全要求

必须实现：

- API slug 参数校验，只允许字母、数字、短横线、下划线和点。
- 路径解析后确认文件位于 `reports/` 目录内。
- 不允许通过 API 读取任意本地文件。
- 不把 Horizon 配置、API Key、日志文件暴露为静态资源。

建议实现：

- 对渲染后的 HTML 做清洗。
- 外链添加 `target="_blank"` 和 `rel="noopener noreferrer"`。
- 对读取失败、解析失败做明确错误响应。

## 15. 后续扩展路线

第一阶段：最小可用版本

- 日报列表。
- 日报详情。
- Markdown 渲染。
- 运行时读取 `reports/`。

第二阶段：阅读体验增强

- 语言筛选。
- 关键词搜索。
- 日期归档。
- 移动端样式优化。

第三阶段：内容能力增强

- 使用 MiniSearch 建立本地搜索索引。
- 支持标签、来源、评分字段。
- 支持 RSS 输出。
- 支持收藏和阅读状态。

第四阶段：数据持久化

- 当需要用户状态、全文索引、访问统计时，再引入 SQLite。
- SQLite 不替代 Markdown 原文，Markdown 仍作为可归档内容源。

## 16. 代码生成执行清单

后续智能体生成代码时，按以下顺序执行：

1. 初始化 Nuxt 3 + TypeScript + pnpm 项目。
2. 创建 `reports/` 并加入示例 Markdown。
3. 安装 `markdown-it`、`gray-matter`、`fast-glob`。
4. 配置 `runtimeConfig.reportsDir`。
5. 实现 `server/utils/markdown.ts`。
6. 实现 `server/utils/reports.ts`。
7. 实现 `GET /api/reports`。
8. 实现 `GET /api/reports/:slug`。
9. 实现首页 `/`。
10. 实现详情页 `/reports/[slug]`。
11. 拆分 `components/reports/*`。
12. 补充基础样式和 Markdown 正文样式。
13. 运行 `pnpm build` 验证生产构建。
14. 运行本地服务并确认新增 Markdown 不需要重新构建即可访问。

## 17. 明确不做

初始版本不要做：

- 不做数据库。
- 不做用户系统。
- 不做 Horizon 采集逻辑。
- 不在 Web 请求中运行 AI 模型。
- 不做静态站点生成作为默认模式。
- 不把 Markdown 内容在构建时打包进前端。

这些功能只有在基础阅读服务稳定后再评估。
